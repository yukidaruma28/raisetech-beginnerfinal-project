## ユースケース詳細

### UC-01: ボードを閲覧する

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | アプリが起動している |
| 事後条件 | ボード画面にすべてのステータス列・問い合わせ・プロパティが表示されている |

**基本フロー**
1. ユーザーがブラウザでアプリにアクセスする
2. システムが `GET /api/inquiries` を呼び出す
3. システムがスケルトン表示でローディング状態を示す
4. データ取得完了後、ボード画面にステータス列と問い合わせカードを表示する

**代替フロー**
- 2-a: データ取得に失敗した場合、エラートーストを表示する

---

### UC-02: 問い合わせを作成する

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | ボード画面が表示されている、ステータスが1件以上存在する |
| 事後条件 | 新しい問い合わせが指定したステータス列の末尾に追加されている |

**基本フロー**
1. ユーザーが画面右上の「新規問い合わせ」ボタン、またはステータス列下部の「＋」ボタンをクリックする
2. 問い合わせ作成モーダルが開く
3. ユーザーがタイトル（必須）・本文・カテゴリ・担当者・ステータス・優先度・ラベルを入力する
4. ユーザーが「作成」ボタンをクリックする
5. システムが `POST /api/inquiries` を呼び出す
6. 新しい問い合わせが該当ステータス列の末尾に表示される

**代替フロー**
- 3-a: タイトルが空の場合、保存ボタンを非活性にする
- 2-a: ×ボタン・オーバーレイクリック・Escキーでモーダルを閉じる
- 1-a: ステータス列下部の「＋」ボタンから開いた場合、ステータスはその列で初期選択される

---

### UC-03: 問い合わせ詳細を編集する

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | 問い合わせが1件以上存在する |
| 事後条件 | 問い合わせの内容（タイトル・本文・カテゴリ・担当者・優先度・ラベル）が更新されている |

**基本フロー**
1. ユーザーが問い合わせカードをクリックする
2. InquiryModal が開き、現在の内容が表示される
3. ユーザーが各項目を編集する
4. ユーザーが「保存」ボタンをクリックする
5. システムが `PATCH /api/inquiries/{id}` を呼び出す
6. InquiryModal が閉じ、カードサムネイルに更新内容が反映されている

**代替フロー**
- 3-a: タイトルを空にした場合、保存ボタンを非活性にする
- 2-a: ×ボタン・オーバーレイクリック・Escキーでモーダルを閉じる（未保存の場合は確認ダイアログを出す）

---

### UC-04: 問い合わせを削除する

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | InquiryModal が開いている |
| 事後条件 | 問い合わせが削除され、InquiryModal が閉じている |

**基本フロー**
1. ユーザーが InquiryModal 内の削除ボタンをクリックする
2. 確認ダイアログが表示される（「この問い合わせを削除しますか？」）
3. ユーザーが「削除」を選択する
4. システムが `DELETE /api/inquiries/{id}` を呼び出す
5. InquiryModal が閉じ、問い合わせがステータス列から消える

**代替フロー**
- 3-a: ユーザーが「キャンセル」を選択した場合、削除を行わずダイアログを閉じる

---

### UC-05: 問い合わせを DnD で移動する

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | 問い合わせが1件以上存在する |
| 事後条件 | 問い合わせが指定したステータス列・位置に移動し、ステータスと position が更新されている |

**基本フロー**
1. ユーザーが問い合わせカードをドラッグする
2. ドラッグ中、ドロップ先にプレースホルダーが表示される
3. ユーザーが目的のステータス列・位置にドロップする
4. システムが `PATCH /api/inquiries/{id}/move` を呼び出す（新しい status_id と position）
5. 問い合わせが新しい位置に表示される

**代替フロー**
- 3-a: 元の位置にドロップした場合、位置は変わらず API は呼び出さない

---

### UC-06: ステータスを作成する

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | 設定画面、またはボード画面のステータス追加ボタンが利用可能 |
| 事後条件 | 新しいステータス列がボードの末尾に追加されている |

**基本フロー**
1. ユーザーがボード右端の「＋ステータスを追加」ボタンをクリックする
2. ステータス作成フォームが表示される（名前・色を入力）
3. ユーザーが名前と色を入力して作成ボタンを押す
4. システムが `POST /api/statuses` を呼び出す
5. 新しいステータス列がボードの末尾に表示される

**代替フロー**
- 3-a: 名前が空の場合、作成ボタンを非活性にする
- 1-a: ステータス数が上限（20件）に達している場合、追加ボタンを非表示にする

---

### UC-07: ステータスを編集する

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | ステータスが1件以上存在する |
| 事後条件 | ステータスの名前・色・並び順が更新されている |

**基本フロー**
1. ユーザーがステータス列のヘッダー（名前部分）をクリックする
2. 名前がインライン編集モードに切り替わる（色変更は別メニュー）
3. ユーザーが新しい名前を入力し Enter を押す
4. システムが `PATCH /api/statuses/{id}` を呼び出す
5. ステータス名が更新された状態で表示される

**代替フロー**
- 3-a: 名前が空の場合、保存せず元の名前に戻す
- 3-b: Esc キーを押した場合、変更を破棄して元の名前に戻す
- 並び替え：ステータス列ヘッダーをドラッグしてボード内で左右に並び替える → `PATCH /api/statuses/{id}/move`

---

### UC-08: ステータスを削除する

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | ステータスが2件以上存在する（最後の1件は削除不可） |
| 事後条件 | ステータスが削除され、所属していた問い合わせは指定された移動先ステータスに移っている |

**基本フロー**
1. ユーザーがステータス列ヘッダーの削除ボタンをクリックする
2. 確認ダイアログが表示される（「このステータスを削除しますか？所属する問い合わせの移動先を選択してください」）
3. ユーザーが移動先ステータスを選択し「削除」を押す
4. システムが `DELETE /api/statuses/{id}?move_to={otherId}` を呼び出す
5. ステータス列がボードから消え、問い合わせは指定した列に移動している

**代替フロー**
- 3-a: ユーザーが「キャンセル」を選択した場合、削除を行わずダイアログを閉じる
- 1-a: 該当ステータスに問い合わせが0件の場合、移動先選択は不要（単純削除）
- 1-b: ステータスが最後の1件の場合、削除ボタンを非活性にする

### UC-13: 問い合わせにラベルを付与する

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | ラベルが1件以上存在する、InquiryModal が開いている |
| 事後条件 | 問い合わせに選択したラベルが追加されている |

**基本フロー**
1. ユーザーが InquiryModal 内のラベル一覧から未選択のラベルをクリックする
2. システムが `POST /api/inquiries/{id}/labels/{labelId}` を呼び出す
3. ラベルがチェック済みになり、カードサムネイルにバッジが表示される

---

### UC-14: 問い合わせからラベルを解除する

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
