## ユースケース詳細

### UC-01: ボードを閲覧する

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | アプリが起動している |
| 事後条件 | ボード画面にすべてのステータス列・問い合わせ・プロパティが表示されている |

**基本フロー**
1. ユーザーがブラウザでアプリにアクセスする
2. システムが `GET /api/inquiries` を呼び出す
3. システムがスケルトン表示でローディング状態を示す
4. データ取得完了後、ボード画面にステータス列と問い合わせカードを表示する

**代替フロー**
- 2-a: データ取得に失敗した場合、エラートーストを表示する

### UC-02: 問い合わせを作成する

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | ボード画面が表示されている、ステータスが1件以上存在する |
| 事後条件 | 新しい問い合わせが指定したステータス列の末尾に追加されている |

**基本フロー**
1. ユーザーが画面右上ヘッダーの「+ 新規」ボタンをクリックする
2. 問い合わせ作成モーダルが開く（ステータスは先頭列、優先度は「低」が初期選択）
3. ユーザーがタイトル（必須）・本文・ステータス・優先度を入力する
4. ユーザーが「作成」ボタンをクリックする
5. システムが `POST /api/inquiries` を呼び出す
6. モーダルが閉じ、新しい問い合わせが該当ステータス列の末尾に表示される

**代替フロー**
- 3-a: タイトルが空の場合、フィールド下部に「タイトルは必須です」を表示し送信を阻止
- 2-a: ×ボタン・オーバーレイクリック・Escキーでモーダルを閉じる
- 5-a: サーバが 422 を返した場合、`details` をフィールド単位の赤帯と各 input のリングで表示しモーダルは閉じない

> **MVP スコープ外（後続 Issue へ）**
> - 各ステータス列下部の「＋ 追加」ボタンは未実装。ヘッダー「+ 新規」のみが起点。
> - category / assignee / labels の入力欄は提供しない（`docs/tech-stack.md` 参照）。
> - 楽観的更新は導入しない（UC-05 DnD 移動で初導入予定）。

### UC-03: 問い合わせ詳細を編集する

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | 問い合わせが1件以上存在する |
| 事後条件 | 問い合わせの内容（タイトル・本文・カテゴリ・担当者・優先度・ラベル）が更新されている |

**基本フロー（Trello / Linear 風 inline-edit + auto-save）**
1. ユーザーが問い合わせ行をクリックする（行全体がクリック判定。checkbox / 優先度アイコンは除外）
2. InquiryEditDialog が開き、現在の内容が表示される
3. タイトル / 本文をクリック → その場で input / textarea に切り替わる
4. フィールドの外をクリック（blur）→ 値が変わっていれば `PATCH /api/inquiries/:id` をフィールド単位で送信
5. ステータス / 優先度の select は変更（change）と同時に即 PATCH
6. リスト側の表示が `invalidateQueries(['inquiries'])` で更新される

**代替フロー**
- 3-a: フィールド編集中に Esc → 編集前の値に戻して閲覧モードに復帰（PATCH しない）
- 4-a: タイトル空 / 256 文字超など zod 検証で弾かれた場合、編集モードを維持しフィールド下に赤エラー表示
- 4-b: サーバ 422 が返った場合、`details` をフィールド下にインライン表示し編集モード維持
- 2-a: ×ボタン / オーバーレイクリック / Esc（編集していないとき）でモーダルを閉じる
- 「保存」ボタンは持たない（auto-save なので未保存状態が発生しない）

> **MVP スコープ外（後続 Issue へ）**
> - 削除ボタン（UC-04 で詳細モーダル内に追加予定）
> - status_id 変更時の `position` 再計算は UC-05（DnD 移動）で対応。今回は status を変えても position は据え置き。
> - 楽観的更新は導入しない（UC-05 で初導入予定）。

### UC-04: 問い合わせを削除する

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | InquiryModal が開いている |
| 事後条件 | 問い合わせが削除され、InquiryModal が閉じている |

**事前条件**: 行をクリックして InquiryEditDialog が開いている

**基本フロー**
1. ユーザーが編集モーダル右上のゴミ箱ボタン（「削除」）をクリックする
2. 確認ダイアログが編集モーダル上にスタックして表示される（「TASK-N を削除しますか？」）
3. ユーザーが「削除」を選択する
4. システムが `DELETE /api/inquiries/:id` を呼び出す（成功 204）
5. 確認ダイアログと編集モーダルが両方閉じ、リストから該当行が消える（`invalidateQueries(['inquiries'])`）

**代替フロー**
- 3-a: 「キャンセル」/ Esc / オーバーレイクリックで確認ダイアログを閉じる（削除しない、編集モーダルは開いたまま）
- 4-a: サーバが 404 を返した場合（他クライアントで先に削除済み）、目的達成として両モーダルを閉じてリスト再取得
- 4-b: ネットワーク障害などその他のエラーは編集モーダル上部の赤帯で通知（モーダルは開いたまま）

### UC-05: 問い合わせを DnD で移動する

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | 問い合わせが1件以上存在する |
| 事後条件 | 問い合わせが指定したステータス列・位置に移動し、ステータスと position が更新されている |

**基本フロー**
1. ユーザーが行の左端のドラッグハンドル（lucide `GripVertical`）にカーソルを合わせる
2. ハンドルをドラッグ → 元の行が半透明、ドロップ先にプレースホルダーが表示
3. ユーザーが目的のステータス列・位置にドロップする
4. UI は楽観的に即時並び替え（vue-query `setQueryData`）
5. 裏で `PATCH /api/inquiries/:id/move` を呼び出し（body: `{ statusId, position }`、`position` は 1-indexed）
6. サーバはトランザクション内で元 status / 新 status の inquiries を **dense int 採番**（1, 2, 3, ...）で一括再計算し、影響を受ける行の position を `update_columns` で更新
7. 成功したら `invalidateQueries(['inquiries'])` で最終整合（万一ずれてもサーバ値で上書き）

**代替フロー**
- 3-a: 元の位置にドロップ（同列内 oldIndex === newIndex）→ API を呼ばない（noop）
- 5-a: PATCH が失敗した場合、楽観的更新を rollback して **snap back**。Toast は出さず、`console.warn` でデバッグ可能にしておく
- 7-a: 同時編集の競合制御は MVP スコープ外（シングルユーザー前提）

> **MVP スコープ外（後続 Issue へ）**
> - Status 列ヘッダーの DnD 並び替え（UC-07 で別途）
> - Toast 通知 / Undo
> - キーボードによるドラッグ（aria-grabbed 等）

### UC-06: ステータスを作成する ✅ 実装済み

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | ボード末尾に「+ ステータスを追加」ボタンが利用可能 |
| 事後条件 | 新しいステータス列がボードの末尾に追加されている |

**基本フロー**
1. ユーザーがボード末尾の「+ ステータスを追加」ボタンをクリックする
2. モーダルダイアログが開き、名前入力欄と色選択 UI（プリセット 8 色 + HEX 直接入力）が表示される
3. ユーザーが名前と色を入力して「作成」ボタンを押す
4. システムが `POST /api/statuses` を呼び出す（`position` はサーバ側で `MAX(position) + 1` を自動採番）
5. ボードのキャッシュが invalidate され、新しいステータス列が末尾に表示される

**代替フロー**
- 3-a: 名前が空 / 色が HEX 形式違反の場合、submit ボタンを非活性にする（zod スキーマ）
- 3-b: サーバ側 422 が返った場合、フィールド下にエラー文言を表示し再入力を促す
- 1-a: ステータス数が上限（10 件）に達している場合、「+ 追加」ボタンを非表示にして案内文に差し替える（バックエンドは `Status::MAX_COUNT` で 422 を返す多重防御）

### UC-07: ステータスを編集する

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | ステータスが1件以上存在する |
| 事後条件 | ステータスの名前・色・並び順が更新されている |

**基本フロー**
1. ユーザーがステータス列のヘッダー（名前部分）をクリックする
2. 名前がインライン編集モードに切り替わる（色変更は別メニュー）
3. ユーザーが新しい名前を入力し Enter を押す
4. システムが `PATCH /api/statuses/{id}` を呼び出す
5. ステータス名が更新された状態で表示される

**代替フロー**
- 3-a: 名前が空の場合、保存せず元の名前に戻す
- 3-b: Esc キーを押した場合、変更を破棄して元の名前に戻す
- 並び替え：ステータス列ヘッダーをドラッグしてボード内で左右に並び替える → `PATCH /api/statuses/{id}/move`

### UC-08: ステータスを削除する ✅ 実装済み

| 項目 | 内容 |
|------|------|
| アクター | ユーザー |
| 事前条件 | ステータスが1件以上存在する |
| 事後条件 | ステータスが削除され、所属していた問い合わせは指定された移動先ステータスに移っている |

**基本フロー**
1. ユーザーがステータス列ヘッダーにカーソルを合わせる（hover で右端にゴミ箱アイコンが現れる）
2. ゴミ箱アイコンをクリックする
3. 確認ダイアログが表示される
   - 所属 Inquiry が 0 件 → 「このステータスを削除しますか？」のみ
   - 所属 Inquiry が >0 件 → 「N 件の問い合わせを別ステータスへ移動して削除します」+ 移動先 select（自分以外の statuses）
4. ユーザーが移動先（必要時）を選択し「削除」を押す
5. システムが `DELETE /api/statuses/{id}?move_to={otherId}` を呼び出す（0 件の場合は `?move_to=` 省略）
6. サーバはトランザクション内で所属 Inquiry を移動先 status に付け替え、移動先内の `position` を dense int で再採番してから当該 Status を destroy する
7. ステータス列がボードから消え、`invalidateQueries(['statuses'])` / `invalidateQueries(['inquiries'])` でリスト再取得される

**代替フロー**
- 3-a: ユーザーが「キャンセル」/ Esc / オーバーレイクリックで削除を中止する
- 5-a: 所属 Inquiry がある状態で `move_to` を欠いた場合、サーバは 409 CONFLICT を返す（API 経由で誤呼び出しされた場合の保険、UI からは到達しない）
- 5-b: `move_to` が存在しない / 自分自身を指す場合、サーバは 404 / 422 を返す。ダイアログ内の赤帯にエラー文言を表示し、削除はキャンセルされる

> **MVP 仕様**
> - 「最後の 1 件は削除不可」のような下限制約は設けていない（要件定義書および本 Issue #32 の仕様）。シングルユーザー前提のため、誤って 0 件にしても再度 UC-06 で追加できる。
> - ソフトデリート / Undo は導入しない。

### ~~UC-09: 優先度を作成する~~ ❌ MVP スコープ外

優先度は **3 段階固定運用（高/中/低）** とするため、追加機能は実装しない。`level` の UNIQUE 制約と 1..3 の範囲制約により、新規追加は DB 設計上も想定外。将来段階数を増やす場合は migration / model 制約 / 優先度アイコンの再設計を伴う別 Issue として扱う。

---

### ~~UC-10: 優先度を編集する~~ ❌ MVP スコープ外

3 段階固定の前提では「高/中/低 の name や color の編集」しか余地がなく、UI を提供する実用的価値が薄いため見送る。色やラベルの調整が必要になった場合は seed / マイグレーションで対応する。

---

### ~~UC-11: 優先度を削除する~~ ❌ MVP スコープ外

3 件固定で常に Inquiry がひもづく前提（`inquiries.priority_id` は NOT NULL、FK は RESTRICT）のため、削除を許す UI は提供しない。

### UC-15: ラベルを削除する

| 事前条件 | ラベルが1件以上存在する |

**基本フロー**
